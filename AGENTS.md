# Строгие правила разработки

## Приоритет правил
- Сначала следовать правилам из AGENTS.md, все расхождения и отступления от них должны быть явно оговорены и задокументированы.

## Типизация
- Запрещено использование типа any. Всегда использовать конкретные типы.

## Структура проекта
- Строго придерживаться структуры папок Nuxt версии 4.
- Правильно организовывать папки и компоненты в каждой директории Nuxt версии 4.

## Зависимости
- Запрещено устанавливать новые зависимости без явного разрешения пользователя.

## Переменные окружения
- Переменные окружения не должны попадать на клиент. Секреты (токены, ключи API, идентификаторы) хранить только в серверном runtimeConfig (без public). Обращение к внешним API выполнять только с сервера (server/api, server utilities).

## Пользовательский интерфейс
- Интерфейс должен быть адаптивным и доступным для всех пользователей и устройств.
- Всегда использовать скелетон при ожидании данных от сервера или внешних сервисов.
- Замени все стандартные элементы управления вводом данных, включая текстовые поля, загрузку файлов, ползунки, выбор цвета, чекбоксы и переключатели, на собственные компоненты, полностью сохранив визуальный стиль проекта.

## Анимации
- Все анимации должны быть быстрыми и плавными.
- Создавать анимации только средствами CSS (transition, animation, @keyframes).

## Стилизация
- В приоритете использовать TailwindCSS.
- Использовать цвета только из палитры в tailwind.config.js. Запрещены произвольные hex-значения и цвета вне палитры.

## Журнал изменений
- Логировать все значимые изменения в CHANGELOG.md.

## Иконки
- Запрещено использовать SVG иконки. Использовать только иконки Remix Icon.

## Документация и макет
- Строго следовать документации в каталоге docs и шаблону в каталоге templates.
- Верстка ориентируется на шаблон из templates. При расхождениях приоритет у визуального референса.

## Рендеринг и индексация
- Не использовать серверный рендеринг. Приложение работает только на клиенте.
- SEO не используется. Индексация сайта поисковыми системами запрещена.

## Хранение данных
- Использовать куки для хранения данных на стороне клиента.
- Использовать куки для работы с пользовательскими данными, настройками и сессиями.
- Не хранить в куки чувствительные данные в открытом виде (пароли, токены, персональные идентификаторы).

## Импорты
- Использовать автоматический импорт Nuxt для всех установленных модулей. Явные импорты не требуются.

## SEO и мета‑теги
- SEO‑мета и open graph мета задавать через `useSeoMeta`, а не через ручные `<meta>` в шаблонах.
- Глобальные дефолты (общий title/description/robots, базовые OG) задавать на уровне лейаута, а переопределения — в страницах.
- Значения, зависящие от данных, устанавливать только после их получения; при ожидании данных показывать skeleton, а мета обновлять реактивно.
- Не прокидывать секреты и любые чувствительные значения в мета‑теги (они попадают в клиент и могут индексироваться/кешироваться).
- Для страниц, которые не должны индексироваться, явно задавать `robots: 'noindex, nofollow'`.

## Управление состоянием
- Использовать Pinia для управления состоянием приложения.

## Соглашения по именованию
- Наименования Vue‑компонентов должны использовать CamelCase с заглавной буквы (PascalCase), например: `UserProfile`, `AppHeader`.
- Наименования Pinia‑сторов должны начинаться с `use`, например: `useUser`, `useSettings`.

## Lazy‑loading тяжёлых компонентов и страниц
- Тяжёлые и редко используемые компоненты (модалки, сложные виджеты, административные экраны) должны подключаться через динамический импорт с lazy‑loading.
- Страницы, не относящиеся к критическому пользовательскому пути, должны загружаться лениво, чтобы не замедлять первоначальную загрузку приложения.
- Для лениво загружаемых компонентов и страниц необходимо предусматривать состояние загрузки (skeleton/spinner) и понятное поведение при ошибке загрузки.
- Не допускать ситуаций, когда лениво загружаемый компонент блокирует основной пользовательский сценарий без явного визуального индикатора.

## Работа с большими списками
- Для отображения больших списков (тысячи и более элементов) обязательно использовать виртуализацию или постраничную загрузку (пагинацию), чтобы избежать фризов интерфейса.
- При пагинации необходимо отображать управляемые элементы навигации (следующая/предыдущая страница, количество элементов на страницу) и текущее состояние (номер страницы, общее количество элементов, если доступно).
- Подгрузку данных для списков реализовывать поэтапно (lazy‑loading/инфинит‑скролл) только при наличии чётких индикаторов загрузки и окончания списка.
- Фильтрацию и сортировку больших списков по возможности выполнять на сервере или в специально оптимизированных структурах, избегая тяжёлых операций на каждом рендере.

## Ограничение тяжёлых операций на клиенте
- Для инпутов, которые инициируют сетевые запросы или сложные вычисления (поиск, фильтрация, автодополнение), обязательно использовать дебаунс или троттлинг с разумным интервалом.
- Частоту сетевых запросов необходимо ограничивать: не допускать множественных параллельных запросов при повторных действиях пользователя; при необходимости отменять устаревшие запросы.
- Любые тяжёлые клиентские операции (парсинг больших структур, сложные вычисления, обходы больших массивов) должны быть вынесены из горячих путей рендера (computed, шаблон) и по возможности выполняться асинхронно или на сервере.
- Повторяющиеся операции с одинаковыми входными данными (например, преобразование одних и тех же структур) должны кэшироваться или мемоизироваться, чтобы уменьшить нагрузку на клиент и улучшить отзывчивость интерфейса.
- Переиспользуемые UI‑компоненты должны иметь префикс, согласованный в проекте (например, `BaseButton`, `BaseInput`), чтобы их легко отличать от доменных компонентов.

## Качество кода
- Следовать лучшим практикам разработки.
- Писать минимальный необходимый код.
- Избегать сложных функций. Код должен быть простым и понятным.
- Всегда комментируй участки кода на русском языке.
- Не отключать правила линтера без веской причины и комментария с объяснением.

## Тестирование
- Писать только осмысленные тесты, которые проверяют поведение системы, а не внутреннее устройство реализации.
- Не добавлять новые библиотеки и конфигурации для тестов без явной необходимости, использовать уже установленный стек: `@nuxt/test-utils`, `vitest`, `@vue/test-utils`, `happy-dom`, `playwright-core`.
- Обеспечивать независимость тестов друг от друга: не делиться общим изменяемым состоянием и всегда сбрасывать окружение между сценариями.
- Давать тестам понятные названия: `describe` описывает сущность или сценарий, `it` — ожидаемый результат, без необходимости читать реализацию.
- Структурировать файлы так, чтобы в одном файле описывались только близкие по смыслу сценарии, а один тест проверял один ожидаемый результат.
- Для повторяющихся шагов внутри тестов использовать вспомогательные функции в рамках файла, а не дублировать код.
- Для модульных тестов использовать лёгкую среду без лишних надстроек, а Nuxt‑окружение подключать только там, где оно действительно нужно (страницы, сложные компоненты, плагины).
- В Nuxt‑тестах избегать прямой работы с глобальными объектами, использовать предоставленные помощники и явно восстанавливать глобальное состояние в конце теста или в блоках подготовки/очистки.
- Подменять внешние зависимости и API на заглушки там, где реальные вызовы усложняют или замедляют тесты (сеть, таймеры, наблюдатели, хранилища).
- Для Nuxt‑автоимпортов и компонентов использовать специализированные средства мокирования, а не ручные хаки, и всегда возвращать моки в исходное состояние между тестами.
- В e2e‑тестах фокусироваться на критических пользовательских путях (запуск приложения, ключевые страницы, формы) и придерживаться принципа «один тест — один пользовательский сценарий».
- Не смешивать в одном файле e2e‑подход и тесты, работающие напрямую с Nuxt‑окружением, чтобы избежать сложных для отладки гибридных сценариев.
- Избегать «хрупких» тестов, завязанных на несущественные детали верстки, и следить за временем выполнения: тяжёлые сценарии выносить в отдельные e2e‑наборы.
- При изменении поведения приложения сначала обновлять или дополнять тесты, а не отключать их, и воспринимать падение тестов как сигнал о реальной проблеме.
- Для локальной проверки запускать только нужный набор тестов (unit, nuxt, e2e), а в пайплайнах — полный набор с разумным разделением по типам.

## Отладка
- Выводить отладочные логи для выявления проблем в процессе разработки.
- Обязательно удалять все отладочные логи после решения проблемы и перед коммитом изменений.
- Оставлять только необходимые логи для обработки ошибок
- Записывай обычные сообщения, предупреждения и ошибки прямо в общий поток вывода системы, чтобы они автоматически отображались в логах без создания отдельных файлов.
- Все ошибки, возникающие в коде, должны приводить к понятному сообщению для пользователя и структурированному сообщению для логов.

## Валидация и формы
- Ошибки валидации всегда отображать пользователю рядом с полем ввода и логировать в общий поток только в агрегированном виде.
- Не отправлять форму на сервер, если клиентская валидация не пройдена.
- Для защиты форм использовать механизмы ограничения частоты отправки, защиты от повторных кликов и других автоматизированных атак.

## Безопасность
- Не использовать v-html без строгой причины и обязательной санитизации данных для защиты от XSS.

## Доступность (a11y)
- Все интерактивные элементы должны быть доступны с клавиатуры и иметь корректный tabindex.
- Использовать aria‑атрибуты для сложных компонентов (модалки, дропдауны, кастомные контролы ввода).
- Не полагаться только на цвет для передачи состояния (ошибка, успех, предупреждение).

## Обновления приложения
- При реализации обновления учитывать, что пользователи уже активно пользуются приложением, и их проделанная работа не должна сломаться или исчезнуть.
- Каждое обновление должно быть безопасным как для пользователя, так и для приложения в целом.

## Планирование и отслеживание задач
- Планирование крупных задач: перед началом работы над крупной задачей необходимо сформировать детализированный список чекпоинтов в @tasks, разбив задачу на небольшие, измеримые шаги, и согласовать этот план при необходимости.
- Отслеживание прогресса по чекпоинтам: по мере выполнения работы обязательна оперативная отметка каждого завершённого чекпоинта в @tasks сразу после его завершения, чтобы состояние задачи всегда отражало актуальный прогресс.
- Актуальность планов: при изменении объёма или приоритетов крупной задачи нужно своевременно обновлять список чекпоинтов в @tasks (добавлять, уточнять или помечать как отменённые), фиксируя реальные договорённости.
