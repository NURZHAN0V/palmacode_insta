# Тесты проекта
+
+ В проекте используется стек `@nuxt/test-utils`, `vitest`, `@vue/test-utils`, `happy-dom`, `playwright-core`. Новые зависимости для тестирования не добавляем без явного решения.
+
+ ## Структура папки `tests/`
+
+ - `tests/unit/` — модульные тесты без Nuxt‑окружения (чистые функции, утилиты, простые компоненты).
+ - `tests/nuxt/` — тесты, которым нужно Nuxt‑окружение (страницы, сложные компоненты, плагины).
+ - `tests/e2e/` — сквозные сценарии (навигация, формы, критические пользовательские пути).
+
+ Имена файлов тестов должны заканчиваться на `.test.ts` или `.spec.ts`, например:
+
+ - `tests/unit/useFormatPrice.test.ts`
+ - `tests/nuxt/AppHeader.nuxt.spec.ts`
+ - `tests/e2e/auth-flow.test.ts`
+
+ ## Общие правила написания тестов
+
+ - Проверяем поведение, а не внутреннюю реализацию.
+ - Тесты не делят общее изменяемое состояние и не зависят друг от друга.
+ - В одном файле описываем только близкие по смыслу сценарии.
+ - `describe` описывает сущность или сценарий, `it` — ожидаемый результат.
+ - Один тест — один ожидаемый результат; сложные случаи разбиваем на несколько тестов.
+
+ ## Использование Nuxt‑окружения
+
+ - Для модульных тестов используем лёгкую среду без лишних надстроек.
+ - Nuxt‑окружение подключаем только там, где оно действительно нужно.
+ - В Nuxt‑тестах не трогаем напрямую глобальные объекты, используем предоставленные помощники.
+ - Любые изменения глобального состояния явно сбрасываем в конце теста или в блоках подготовки/очистки.
+
+ ## Моки и заглушки
+
+ - Внешние API, сеть, таймеры, наблюдатели и хранилища по возможности подменяем заглушками.
+ - Для Nuxt‑автоимпортов и компонентов используем специализированные средства мокирования.
+ - Всегда возвращаем моки в исходное состояние между тестами.
+
+ ## E2E‑подход
+
+ - Фокус на критических пользовательских путях (запуск приложения, ключевые страницы, формы).
+ - Один тест — один пользовательский сценарий.
+ - Не смешиваем в одном файле e2e‑подход и тесты, работающие напрямую с Nuxt‑окружением.
+
+ ## Запуск тестов
+
+ - Базовый запуск через Nuxt: `npx nuxt test` (ищет файлы `*.test.*` и `*.spec.*` по проекту).
+ - Для локальной проверки запускаем только нужный набор (unit, nuxt, e2e), когда появится отдельная конфигурация под проекты Vitest.
+ - В CI запускаем полный набор тестов с разделением по типам.
